About these flows

These are the MVP user flows for Vitolero, a camera-first cigar app. The app opens to the camera. No sign-up wall. Three features: Snap & Share, Pairing, and Journal. Post-MVP flows and open questions live in separate sub-pages.

Navigation rules · resolved for MVP

Two pips, one boolean. Camera pip (bottom-right) appears on every screen except the camera itself and screens where the camera is the implicit next action (sommelier, cigar card, filter preview, share). Journal pip (top-right) is hidden on the very first session and unlocks the moment the first identification auto-saves. Account creation stays deferred. First Journal tap triggers Google sign-in (PRD §6.3); no upfront wall.

Pattern lineage: Seek (journal-unlocks-after-first-capture), Yuka (anonymous-first), Snapchat (camera-as-home, no chrome). See strategy/navigation-plan.md for the full state/affordance matrix and pattern catalog.

Section 1 · MVP

First-time User

Per the PRD §6.1. Camera is the default state on app open. No sign-up wall. Identification, then share. Pairing is reachable from the share screen or identification result. Journal pip is hidden on the first session. It unlocks the moment the first identification auto-saves. The first tap on the Journal pip triggers Google sign-in (per §6.3).

Design principles: No "Open Scanner" splash. The camera is the home. No manual brand picker: identification surfaces automatically at 80%+ confidence, with up to 3 candidates if ambiguous. No taste-profile hook: the share is the primary CTA after identification. Save is local (account-less) until the user opens Journal.

Section 2 · MVP Hero

Snap & Share

The hero feature. Per the PRD §4.1, every interaction in the app starts here. Camera default state, photo capture, sommelier-style processing, automatic brand-and-line identification (no vitola), filter selection from 3-5 cigar-specific overlays, magazine-style share card, then handoff to Facebook (deep link via sharer.php) or other platforms (Web Share API).

MVP specs: identification under 10s, 80%+ confidence threshold, up to 3 candidates if ambiguous, brand and line only (vitola deferred: full vitola detection pushed processing to 40-50s). Filter pipeline desaturates the background and keeps the cigar saturated; a semi-transparent strip overlay at the bottom carries brand, line, and a subtle Vitolero mark. Magazine-quality, not screenshot-quality. One platform per share. After share, the user leaves the app (post-share reacquisition is acknowledged as an open problem).

Section 3 · MVP

Pairing

Per PRD §4.2. A separate journey from filters and overlays. Photo quality is irrelevant once identification has happened. Input: a cigar line. Output: exactly 6 items, 3 food and 3 drink. One-directional only: cigar → food/drink. The cigar is the hero (big, vertical) with the pairings listed beside it; item names stand alone, no category tags or icons. Forward navigation always visible (no dead end).

MVP specs: 10-15s latency from cigar line input to 6-item result. Layout: large vertical cigar (~80% height) beside a compact pairing list. Per-item category tags and icons removed; Food/Drink dividers de-emphasized. One suggestion highlighted as the featured pairing. After viewing, the user can snap a new cigar or open the Journal. No dead end. Final pairing navigation owned by Dimitar + Fabio.

Section 4 · MVP

Journal

Per the PRD §4.3. A persistent visual gallery of the user's cigar photos. Every cigar identified is auto-saved to the local Journal. No explicit save action. The user can delete entries later. The first tap on Journal triggers Google sign-in to sync across devices. Pinterest-style masonry (or Instagram-style grid, layout choice still open). Each entry shows the filtered photo and a cigar card with product information.

Auto-save model: the cigar enters the Journal the moment identification surfaces. No "save?" prompt: it's already there. Delete is the inverse action, available from the Journal entry detail.
Schema requirement: the Journal data structure must support future recommendations from day one. No rebuild when Phase 2 ships. No recommendation UI in MVP, no recommendation logic in MVP, just a schema that fits both.
Pre-account storage: photos persist locally before sign-in. Local storage is not 100% reliable across devices. The first Journal tap converts that fragile local store into durable cloud storage via Google sign-in.

Section 5 · MVP

Cigar Detail Screen

Per the PRD §4.4. The full-data view for a single cigar. Accessible from two entry points: the Cigar Card (via "Details" link after identification) and any Journal entry (tap the cigar card). The identification result card stays lean: brand, line, one-line flavour. This screen is the destination for users who want the complete picture.

Fields (in order): Brand logo · Brand name · Line / vitola · Date smoked · Wrapper · Binder · Filler · Strength · Smoke time · Flavours (first 3 chips; expand for more) · Description paragraph. All read-only in MVP. Back navigation returns to the entry point.

Section 6 · MVP

Returning User

Per the PRD §6.2. The app does not open to the camera on every visit. Only on a fresh first open. On reopen, the app shows the last photo taken with action buttons available: Share, Pair, Journal, and a button to snap a new cigar. From there the user can re-engage with a previous result, start a fresh capture, or open Journal.

Open problem (PRD §6.3): after the user shares and exits the app, there is currently no mechanism to bring them back. Facebook deep links don't return the user; Web Share API has no callback for image-file shares; PWA push notifications are severely limited on iOS. Snap frequency is the leading indicator to watch. Fabio is investigating technical options. Journal activation creates a more durable hook beyond the next smoke.

Section 7 · MVP

Install to Home Screen

Per PRD §4.5. A lightweight prompt that turns first-time visitors into returning users without ever blocking the camera-first flow. Shown only after the first successful identification (never on first open), at most twice. The installed app on the home screen is the most durable surface Vitolero has after a share, so the prompt sells the value first, then guides the install in the fewest possible steps. Split by platform because iOS and Android behave differently.

MVP specs: Trigger: after the first successful identification (first share or first Journal save, whichever comes first). Frequency: at most twice, then never again. iOS (Safari): no beforeinstallprompt, so a custom explainer points at the Share icon. Android (Chrome): capture the native beforeinstallprompt and fire it on demand. Bottom-sheet only; never a full-screen interstitial; never blocks the camera or share.
